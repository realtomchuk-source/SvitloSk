# Сторінка «Підчерга» (Пошук підчерги за адресою)

Цей автономний модуль реалізує функціонал пошуку підчерги погодинних відключень (ГПВ) за фізичною адресою у межах **Старокостянтинівської міської територіальної громади**.

---

## 🏗️ Архітектура модуля

Модуль спроектований повністю автономним, відповідно до концепції **Offline-First**. Це гарантує стабільну роботу та миттєвий інтерфейс автодоповнення навіть під час повного знеструмлення (блекауту) та відсутності інтернет-зв'язку.

### Схема папок:
```
AddressSearch/
├── AddressSearch.tsx            # Головна сторінка з покроковим вибором (Крок 1 -> Крок 2 -> Крок 3)
├── AddressSearch.module.css     # Кастомні стилі (Vanilla CSS), повністю ізольовані від решти проекту
├── README.md                    # Ця технічна документація
├── components/                  # Допоміжні компоненти UI
│   ├── AutoCompleteInput.tsx    # Спеціалізоване поле введення вулиць/селищ з автопідказками
│   └── MissingAddressForm.tsx   # Лаконічна форма для надсилання запиту на нову адресу
├── data/                        # Локальна база адрес
│   └── starokost_addresses.json # Оптимізована деревоподібна JSON база даних
└── services/                    # Логічні сервіси
    └── addressService.ts        # Парсинг та фільтрація адрес
```

## 🗃️ Структура локальних даних (JSON)

Локальна база адрес [starokost_addresses.json](./data/starokost_addresses.json) побудована у вигляді ієрархічного дерева:
1.  **Рівень 1**: Розділення на `city` (Місто) та `okrugs` (Старостинські округи).
2.  **Рівень 2 (тільки для округів)**: Назва Старостинського округу (наприклад, `Григорівський`).
3.  **Рівень 3 (тільки для округів)**: Назва Села (наприклад, `с. Григорівка`).
4.  **Рівень 4**: Назва Вулиці (наприклад, `вул. Миру`).
5.  **Рівень 5**: Карта номерів будинків та їхніх підчерг:
    *   Якщо вся вулиця належить до однієї підчерги, значенням вулиці може бути рядок (наприклад: `"універсальна": "1.1"`).
    *   Якщо вулиця розділена, значенням є об'єкт будинків: `{"1": "1.1", "2": "1.2", "12-А": "2.1"}`.

## 🔄 Інтеграційні зв'язки сторінки

Хоча сторінка автономна в межах своєї директорії, вона інтегрується в наступні точки додатку SvitloSk:

1.  **Zustand Store (`updateUserConfig`)**:
    При успішному знаходженні адреси:
    *   **Дизайн результату (iOS-Style Consolidated Header)**: Результат пошуку та дані обраної адреси об'єднуються в **єдину горизонтальну картку хедера** (`unifiedHeaderCardOrange` / `unifiedHeaderCardGray`) у верхній частині сторінки:
        *   **Зліва**: Квадратна 3D-плашка з номером підчерги, яка динамічно підсвічується помаранчевим (світло є) або сірим (світла немає). Якщо підчерга потребує уточнення, показується сіра плашка з іконкою знаку питання.
        *   **Справа**: Текстові дані обраної локації з вишуканим вирівнюванням по правому краю (`text-align: right`) та підписом «Обрана адреса».
        *   **Рамки та Тіні**: Синхронізуються за кольором відповідно до статусу живлення (Orange vs Gray) для створення стійкого візуального зв'язку між блоками.
        *   **Фон**: Вся сторінка має світло-сріблясте тло `#F5F5F7` (`var(--system-bg-light)`), на якому білі картки виглядають об'ємно та контрастно.
    *   **Преміальний стек дій**:
        Білі капсульні кнопки дій винесені під хедер як самостійний вертикальний стек на сріблястому тлі:
        *   **Авторизований користувач**:
            Бачить помаранчеві та сірі капсули:
            1.  *«Показувати цю підчергу першою в додатку»* — оновлює `startGroup` у Zustand store та м'яко перенаправляє в `/cabinet`.
            2.  *«Налаштувати пуш-сповіщення для підчерги»* — перенаправляє в `/cabinet` та відразу відкриває форму налаштування сповіщень.
        *   **Гість (анонімний)**:
            Бачить кнопку *«Показувати цю підчергу першою в додатку»* та красивий інформаційний банер із лінком-поясненням про переваги реєстрації (можливість підписки на пуші).
        *   *«Повідомити про відсутню в базі адресу»* — відкриває BottomSheet-форму, яка завдяки `z-index: 2000` повністю перекриває плаваючий Tab Bar (z-index 1000).
        *   *«Почати новий пошук з самого початку»* — скидає покроковий вибір у корінь.

2.  **Supabase (`missing_address_requests`)**:
    У разі відсутності адреси користувач надсилає запит, який фіксується у хмарі Supabase. Форма збору має інтелектуальний пошук: підтягує вже існуючі вулиці локації, але одночасно дозволяє **повністю вписати нову вулицю вручну**.

3.  **Панель Адміністратора (Планується)**:
    Запити на додавання нових адрес будуть доступні в адмінці під новою вкладкою **«Запити адрес»** для швидкого опрацювання (розробка інтерфейсу відкладена до окремої команди користувача).

## 📈 Перспективний план розвитку (Roadmap)

Для технічного масштабування у документацію закладаються наступні майбутні фази:

### Фаза 2: Гібридний Offline-First (Supabase $\leftrightarrow$ IndexedDB)
*   **Як це працюватиме**: База адрес зберігається у великій Supabase-таблиці.
*   **Синхронізація**: Під час першого запуску додатку (або коли пристрій онлайн) локальний Service Worker завантажує оновлення адрес та записує їх у IndexedDB користувача.
*   **Перевага**: Адреси можна редагувати на сервері в будь-який момент (вони оновляться у користувачів «на льоту»), але додаток все одно працюватиме на 100% автономно під час відсутності зв'язку.

### Фаза 3: Геолокаційний пошук (GPS-Link)
*   **Як це працюватиме**: Додавання кнопки **«Визначити мою адресу автоматично»**.
*   **Метод**: Використовуючи HTML5 Geolocation API, додаток бере GPS-координати пристрою користувача і через швидкий реверсивний геокодинг (прямо у Service Worker) пропонує найближчу адресу зі списку.
