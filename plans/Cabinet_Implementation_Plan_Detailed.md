# Детальний План: Відновлення та модернізація сторінки «Особистий кабінет» (Версія 3.2 - Фінальна Деталізація)

## 1. 🎯 Головна Мета
Створити сучасний, швидкий особистий кабінет. Додаток працюватиме за гібридною моделлю:
- **Базовий рівень:** Кожен користувач може зайти в додаток і бачити графік завдяки локальній базі (IndexedDB), це працює миттєво.
- **Преміум рівень (Пуш-сповіщення):** Доступний **ВИКЛЮЧНО** авторизованим через Google користувачам. Логіка сповіщень базується на генерації «Карти пушів на добу» з урахуванням індивідуальних налаштувань слотів та DND (Не турбувати).

## 2. 📅 Детальні Етапи Реалізації (Junior-Ready)

*Важливо: Кожен етап розрахований на послідовну розробку. Не переходьте до наступного етапу, поки поточний не протестовано.*

### Етап 1: Візуальна реставрація та Компонентна структура (The Face)
*Фокус лише на Інтерфейсі (React + CSS Modules + Lucide Icons).*

1. **Очищення та структура `Cabinet.tsx`:**
   - Створити папку `src/pages/Cabinet/components/`.
   - Створити 4 базові компоненти: `ProfileCard.tsx`, `NotificationSlots.tsx`, `GeneralSettings.tsx`, `AboutSection.tsx`.
   - `Cabinet.tsx` стає просто контейнером, який імпортує та виводить ці компоненти.
2. **Стилізація (CSS Modules):**
   - Створити файл `Cabinet.module.css`. Всі класи писати туди (наприклад `.container`, `.card`, `.toggle`). Це захистить від конфліктів класів по всьому додатку.
   - Використати глобальні змінні з `index.css` (напр. `var(--accent)`) для підтримки світлої/темної теми.
   - Реалізувати Glassmorphism: `backdrop-filter: blur(12px); background: rgba(255, 255, 255, 0.5);`.
3. **Іконки:**
   - Повністю видалити старі інлайнові SVG.
   - Імпортувати `lucide-react`: `<ShieldCheck>`, `<Bell>`, `<Clock>`, `<Settings>`, `<Share2>`.
4. **Мокінг даних (Mocking):**
   - Жорстко прописати масив з 2-ма фейковими слотами, щоб переконатися, що верстка (динамічний `map` карток) працює ідеально.
5. **UI-Елементи (Bottom Sheets):**
   - Для пунктів "Про застосунок" та (в майбутньому) форми зворотного зв'язку будемо використовувати спливаючі "шторки" (Bottom Sheets / Modals) замість маршрутизатора (React Router). Це дасть додатку справжнє "нативне" відчуття без перезавантаження сторінок.

### Етап 2: Локальна автономія та Стан (The Heart)
*Підключення існуючого Zustand та IndexedDB.*

1. **Zustand Store (`src/store/useStore.ts`):**
   - Перевірити, чи інтерфейси `UserConfig` та `Slot` відповідають новим вимогам (Підчерга 1-12, DND Start, DND End).
2. **IndexedDB Інтеграція (`services/db.ts`):**
   - Прив'язати перемикачі (наприклад, "Стартова підчерга") до функції оновлення Zustand.
   - Переконатися, що Zustand Middleware автоматично синхронізує ці зміни з локальним `idb` (щоб при оновленні сторінки налаштування не зникали).
3. **Обмеження доступу до Пушів (Важливо!):**
   - Картки слотів сповіщень повинні мати стан "Заблоковано", якщо користувач не залогінений. Додати кнопку "Увійти через Google для налаштування сповіщень".

### Етап 3: Хмарна Авторизація та Синхронізація (The Bridge)
*Підключення Supabase (Тільки для Google Auth).*

1. **Google Auth (Supabase):**
   - Налаштувати OAuth Provider (Google) в панелі Supabase.
   - Реалізувати функцію входу/виходу в `ProfileCard.tsx`.
   - Після входу записувати `uuid`, `email`, та `name` в таблицю `profiles`.
2. **Синхронізація Слотів:**
   - Таблиця `user_push_subscriptions` у Supabase: `user_id`, `slot_id`, `group_id`, `dnd_start`, `dnd_end`, `fcm_token`.
   - Як тільки користувач логіниться і змінює слот, дані миттєво летять у цю таблицю.

### Етап 4: "Карта пушів на добу" (The Voice)
*Налаштування Firebase Cloud Messaging (FCM).*

1. **Створення та налаштування Firebase проекту:**
   - Створити проект у Firebase Console.
   - Згенерувати конфіг та додати обов'язковий Service Worker (`firebase-messaging-sw.js`) у папку `public` для отримання пушів у фоні.
2. **Клієнт (Отримання Токена):**
   - Після успішного Google-логіну, запитати дозвіл у браузері на Push-сповіщення.
   - Отримати FCM Токен і зберегти його у таблицю `user_push_subscriptions` (Supabase).
3. **Тестування та розробка генератора "Карти Пушів" (TDD Підхід):**
   - Логіка створення карти пушів має бути написана як чиста TypeScript функція.
   - *Локальне тестування:* За допомогою `Vitest` створюються мокові дані (фейковий графік відключень та БД юзерів). Функція тестується локально на коректність фільтрації користувачів, які сплять (перевірка DND). Це гарантує 100% надійність логіки.
4. **Розгортання на Бекенді:**
   - Після успішних локальних тестів, перевірений скрипт розгортається як Supabase Edge Function (або cron-задача).
   - *Логіка роботи:* При оновленні графіка скрипт збирає дані юзерів, пропускає їх через нашу функцію та розсилає пуші через Firebase Admin SDK (за 15 та 5 хв до події).

### Етап 5: Фіналізація та Доступність (The Polish)
*Зручність та дрібниці.*

1. **Зворотній зв'язок (`mailto:` + Supabase):**
   - Кнопка відкриває Bottom Sheet або напряму формує посилання типу `mailto:support@svitlosk.com`.
   - Одночасно на `onClick` відправляється запит в Supabase, який записує прапорець `has_feedback: true` для цього користувача (щоб в адмінці було видно, що від нього полетів лист).
2. **Функція "Поділитись":**
   - Підключити `navigator.share()` API. (Якщо працює на ПК, запропонувати `navigator.clipboard.writeText` як фолбек).
3. **UX покращення:**
   - Додати скелетони (Skeleton Loaders) під час завантаження даних та toast-повідомлення ("Налаштування збережено").
